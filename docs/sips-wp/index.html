Secure IPS

May 22, 2021

Secure IPS Overview#

Cisco Secure IPS (NGIPS) provides network visibility, security intelligence, automation, and advanced threat protection. It uses industry-leading Snort intrusion prevention capabilities and multiple techniques to detect and protect against even the most sophisticated network attacks.

Cisco Secure IPS continuously discovers information about the network environment, including data about operating systems, mobile devices, files, applications, and users. It then uses this information to build network maps and host profiles, which provide the contextual information needed for better decisions about intrusion events. This information also includes details for more robust automation of critical threat protection features. One of these features, security automation, correlates intrusion events with vulnerabilities to prioritize critical threats. Another feature, Firepower Rule Recommendations, analyzes weaknesses in a network and recommends appropriate security policies.

Cisco's Talos Security Intelligence and Research Group collect and correlate threats in real-time using the world's largest threat detection network. Their efforts result in vulnerability-focused IPS rules and embedded IP-, URL-, and DNS-based security intelligence for Cisco Secure IPS. Talos is one of the largest commercial threat intelligence teams globally, comprised of world-class researchers, analysts, and engineers. Talos has unrivaled telemetry and sophisticated systems to create accurate, rapid, and actionable threat intelligence for Cisco customers, products, and services. Talos defends Cisco customers against known and emerging threats, discovers new vulnerabilities in common software, and interdicts threats in the wild before they can further harm the Internet at large. Talos maintains the official rule sets of Snort.org, ClamAV, and SpamCop and provides many open-source research and analysis tools.

IPS Deployment#

Depending on the network requirements, there are several choices to deploy an IPS device.

  • Routed mode

  • Transparent mode

  • Inline mode

  • Passive mode.

This section provides an overview of various Cisco Firepower Threat Defense (FTD) device deployment modes.

Firewall Mode#

Firepower Threat Defense devices support two firewall modes for standard firewall interfaces: Routed and transparent firewall modes. The firewall modes only affect standard firewall interfaces.

  • Routed: In routed mode, the FTD device is a router hop in the network with each routed interface on a different subnet.

  • Transparent: In transparent mode, the FTD is a Layer 2 device that acts like a bump in the wire. It does not act as a default gateway or router hop to connected devices, making it easier to introduce into an existing network. The traffic flow between interfaces is still controllable using access control rules with typical firewall checks. The FTD device connects the same network subnet between its interfaces.

IPS-only Mode#

IPS-only mode only supports IPS security policies and bypasses many firewall checks. IPS-only mode is typical when dedicated IPS devices are required. Routed and transparent firewall modes support IPS-only interfaces.

  • Inline: An inline set acts as a bump on the wire and binds two interfaces together to inject into an existing network. Inline interfaces receive all traffic unconditionally, but all traffic received on these interfaces is retransmitted out of an inline set unless explicitly dropped.

  • Inline-tap: Inline-tap mode deploys the IPS device inline, but the network traffic flow is undisturbed because a copy of each packet is sent to the NGIPS. Intrusion rules generate events whenever there is a match, but the device does not block any traffic. Customers use inline-tap deployment to analyze packets that would have dropped in an inline deployment. Based on the results, an administrator can choose to revise the intrusion policy and deploy the updated policy to block malicious traffic without any cable reconfiguration.

  • Passive: Passive interfaces monitor traffic flowing across a network using a switch SPAN or mirror port. The SPAN or mirror port allows copying traffic from other ports on the switch to provide visibility within the network without being in the network traffic flow. When configured in a passive deployment, the system cannot take specific actions such as blocking or shaping traffic. Passive interfaces receive all traffic unconditionally, never retransmitting traffic received on these interfaces.

Resilient IPS Design#

Cisco Secure IPS solutions provide resilient network performance while keeping data safe and securely accessible to legitimate users. FTD uses various Cisco hardware and software-based features to deliver business resiliency and superior threat defense.

  • Hardware bypass: Hardware bypass (also known as fail-to-wire) is a physical layer (Layer 1) bypass that allows paired interfaces to go into bypass mode. The hardware forwards packets between the interface pairs without software intervention to provide network connectivity when software or hardware failures occur. All Cisco Firepower NGIPS threat appliances offer the capability of operating inline via fail-to-wire/bypass network modules.
note

When switching from normal operation to hardware bypass or from hardware bypass back to normal operation, there is the possibility of a few seconds of traffic interruption. Several factors can impact the length of the interruption. These include copper port auto-negotiation, the optical link partner\'s behaviour (e.g., how it handles link faults and de-bounce timing), spanning tree protocol convergence, and dynamic routing protocol convergence. During this time, connections may be dropped, such as application identification errors when analyzing connections midstream after returning to normal operations.

Hardware bypass is only supported in inline mode on a fixed set of ports. For example, the pairing of port 1 with port 2 or Port 3 with Port 4 is supported. However, Port 1 cannot pair with Port 4.

  • Passive Monitoring: Offers a simple approach to gain visibility into network traffic using a network tap to send a copy of the traffic to the Firepower IPS appliance. Taps operate nearly anywhere in a network environment without interrupting the regular traffic flow. Network taps are unobtrusive and not detectable by the network. However, this added tap is not free and is an extra cost to the business. Additionally, it introduces operational overhead to manage in a large-scale environment but can also be used to swap out appliances if necessary without impacting the network.

  • High-availability: High availability, also called "failover," is a deployment of two identical FTD devices connected through a dedicated failover link and, optionally, a state link. Firepower NGIPS supports Active/Standby failover, where one unit is the active unit passing traffic. The standby unit does not actively pass traffic but instead synchronizes configuration and other state information with the active unit. When a failover occurs, the active unit fails over to the standby unit, becoming active.

Monitoring the active unit's health (hardware, interfaces, software, and environmental status) determines the specific failover conditions to trigger a failover. Cisco recommends that failover links and data interfaces travel through different paths to decrease the chance that all interfaces fail simultaneously. If the failover link is down, the FTD device can use the data interfaces to determine if a failover is required. Subsequently, the suspension of failover operations continues until the health of the failover link is restored.

Scalable IPS Design#

An IPS must be scalable to support the business's future growth and the ever-increasing need to inspect additional traffic without compromising performance or introducing any latency. This section provides an overview of the various ways the Cisco Secure IPS ensures scalability.

  • Cluster: Supported only on the Firepower 9300 and the Firepower 4100 series hardware, clustering allows the grouping of multiple NGIPS units into a single logical device. A cluster provides all the convenience of a single NGIPS device (management, integration into a network) while achieving the increased throughput and redundancy of multiple NGIPS devices.

  • A single IPS cluster can include up to 6 units of any combination. To illustrate, one module in six separate Firepower 9300 chassis, two modules in three separate Firepower 9300 chassis, or any combination to a maximum of six modules. The Firepower 9300 model supports intra-chassis and inter-chassis clustering. The Firepower 4100 series units support inter-chassis clustering using up to six chassis.

    The two diagrams below show Cisco IPS clustering deployment options in a production network environment:

An IPS cluster supports the following deployment scenarios:

  • Inter-Chassis Cluster: For inter-chassis clustering, you must manually assign physical interface(s) to the EtherChannel for communications between chassis via the Cluster Control Link (CCL).

  • Intra-Chassis Cluster: A cluster built with security modules on the same chassis (Firepower 9300 only). For intra-chassis clustering, an internal communication link uses the Firepower 9300 backplane for cluster communications.

  • Inter-site Cluster: For inter-site installations, take advantage of FTD clustering by configuring each cluster chassis to a separate site ID.

Site IDs work with site-specific MAC addresses and IP addresses. Packets sourced from the cluster use a site-specific MAC address and IP address, while packets received by the cluster use a global MAC address and IP address. This feature prevents the switches from learning the same global MAC address from both sites on two different ports, which causes MAC flapping; instead, the switches only learn the site MAC address. Site-specific MAC addresses and IP addresses are supported for NGIPS in routed mode using Spanned EtherChannels only.

Site IDs enable:

  • Flow mobility using LISP inspection

  • Director localization to improve performance and reduce round-trip time latency for inter-site clustering for data centers

  • Site redundancy for connections where a backup owner of a traffic flow is always at a different site from the owner

  • Multi-instance: The multi-Instance capability on the Firepower 4100/9300 lets an IPS run on container instances that use a subset of the security module/engine resources, enabling the isolation of IPS policy management and data traffic inspection.

The maximum number of IPS instance a device can support is dependent on its available hardware resources because each container instance requires dedicated hardware resources assigned to it. Instances use dedicated hardware resources primarily from the security module, but they sometimes need the supervisor's resources.

Firepower appliances support two instance types:

  • Native instance: Native instances are instances of FTD installed on the bare metal hardware of the appliances, and these instances use all the hardware resources available on the appliance.

  • Container instance: A container instance, on the other hand, is an instance that does not consume all the available hardware resources on the appliance. Instead, it only consumes the number of hardware resources as specified by the administrator.

  • VRF: Virtual routing and forwarding (VRF) creates multiple instances of independent routing tables, which allows the usage of the same or overlapping IP addresses without conflicts. FTD allows the same network to be part of two or more virtual routers, which involves applying multiple policies at the interface or the virtual router level. With a few exceptions, overlapping IP addresses do not impact the routing functions and most NGFW and IPS capabilities.

  • IPS with ACI: The Cisco Application Policy Infrastructure Controller (APIC) is a single point of control for centralized functions on the Cisco Application Centric Infrastructure (ACI). The ACI fabric provides for the integration of L4-L7 services as an integral part of an application accomplished using an APIC-managed service graph, which requires a L4-L7 device package. The imported device package exposes configuration parameters in APIC to orchestrate a given configuration onto the device.

    The APIC can automate the insertion of services such as a Cisco Firepower Threat Defense (FTD) northbound between applications, also called endpoint groups (EPGs). The APIC uses northbound Application Programming Interfaces (APIs) for configuring the network and services. APIs are used to create, delete, and modify configurations with managed objects.

    To configure and monitor service devices, the APIC requires a device package. A device package manages a class of service device and provides the APIC with information about the device so that the APIC knows the device functions. A device package enables the insertion and configuration of network service functions on a service device, such as an FTD appliance.

    The FTD Fabric Insertion (FI) Device Package uses a hybrid model (Service Manager, in ACI terminology) to share the responsibility of the full-device configuration between security and network administrators:

    Security administrator: Uses the FMC to pre-define a security policy for the new service graph, leaving Security Zone criteria unset. The new policy rule(s) define the appropriate access (allowed protocols) and the threat components such as IPS, malware, URL filtering, Threat Grid, and more.

    Network administrator: Uses the APIC to orchestrate a service graph, insert an FTD device into the ACI fabric, and attach directed traffic pre-defined security policies. Inside the APIC's L4-L7 Device Parameters or Function profile, the network administrator sets parameters defined in this guide, including matching a pre-defined FMC Access Control Policy and Rules.

  • When the APIC matches the name of the Access Control Policy Rule in the FMC, it simply inserts newly created security zones into the rule(s). If a rule doesn't exist, the APIC creates a new rule by that name, attaches security zones to it, and sets the Action to Deny. This action forces the security administrator to update the new rule(s) criteria and the appropriate set of protections before allowing traffic for a given service graph.

IPS Policy Design#

An Intrusion policy is a defined set of intrusion detection and prevention configurations that inspect traffic for anomalies and threats, allowing inline deployments to block or alter malicious traffic. An access control policy invokes intrusion policies as a system's last line of defense before allowing traffic to its destination.

Intrusion rules comprise the core of an intrusion policy. Cisco Firepower Threat Defense (FTD) includes thousands of intrusion rules to provide greater security coverage by default. Furthermore, the FMC supports importing custom rules or the Cisco Talos Security Intelligence and Research Group's new rules to ensure protection from the latest threats. To help design and maintain an intrusion policy with a large ruleset, the FMC includes several built-in tools.

  • Policy Layers: Larger organizations managing many devices may have multiple intrusion and network analysis policies to support the unique needs of different departments, business units, or, in some instances, different companies. Configurations in both policy types are contained in building blocks called "layers," which can facilitate the management of multiple policies.

Layers in intrusion and network analysis policies work in essentially the same way. If layers are not added to a policy when creating the policy, the system automatically includes the changes into a single configurable layer, which is initially named "My Changes." Intrusion policies support up to 200 layers with any combination of settings. They are easy to manage, given the ability to copy, merge, move, delete, and share an individual user layer with other policies of the same type.

  • System-provided Policies: The Firepower system provides several pairs of network analysis and intrusion policies. Using system-provided network analysis and intrusion policies allows a business to take advantage of Talos' expertise and knowledge. Talos defines intrusion and preprocessor rule states and provides the initial configurations for preprocessors and other advanced settings. The system-provided policies are useable as-is or as the base for custom policies.

  • Balanced Security and Connectivity network analysis: Built for both speed and detection, this policy serves as a good starting point for most networks and deployment types. The system uses the Balanced Security and Connectivity network analysis policy as the default.

  • Connectivity Over Security network analysis: Built for networks where connectivity, this policy ensures the ability to get to all resources at all times, which takes precedence over network infrastructure security. This intrusion policy enables far fewer rules than the other system-provided policies. This policy only enables the most critical rules that block traffic.

  • Security Over Connectivity network analysis: A policy built for networks where network infrastructure security takes precedence over user convenience. This intrusion policy enables additional network anomaly intrusion rules that could alert on or drop legitimate traffic.

  • Maximum Detection network analysis: This policy is primarily provided for testing purposes only, given the potential for a significant operational impact on network traffic. For example, this intrusion policy enables rules in many threat categories, including malware, exploit kit, old and common vulnerabilities, and known in-the-wild exploits.

  • Variable Sets: When the system uses an intrusion policy to evaluate traffic, it uses an associated variable set. The variables represent values commonly used in intrusion rules to identify the source and destination IP addresses and ports. Also, variables can represent IP addresses in rule suppression and dynamic rule states. Even when using the system-provided intrusion policies, Cisco strongly recommends modifying the default variable set variables or creating custom variable sets that accurately reflect the network environment. To learn more, read the Intrusion Policy Best Practices guide.

IPS Integration#

The Secure IPS solution supports sending events directly to the Cisco cloud from Firepower Threat Defense (FTD) devices. Specifically, IPS devices send events to the Security Services Exchange (SSE), which are then promoted to incidents that appear in the Cisco SecureX threat response.

The Secure IPS integration with Cisco SecureX threat response uses intrusion alerts from Firepower devices to provide contextual awareness and correlation with additional insights from other security technologies to deliver a curated alerts repository. This integration provides businesses with limited security operations center (SOC) resources to simplify and enhance the investigation of the most critical security incidents.

IPS Migration from IBM to Cisco#

On August 15, 2017, IBM announced their intention to stop selling the IBM QRadar Network Security (XGS) product line. This decision took effect on December 31, 2017, with current customers' support until December 31, 2022. IBM's decision was based on an analysis of conditions, competitiveness, and strategic fit and reflected their belief in the strength and value of their relationship with Cisco Systems Security Business Group (SBG). One aspect of this relationship is that IBM's sales organizations introduce Cisco's Secure IPS and Secure Firewall solutions to XGS/GX customers, where appropriate, who are looking to refresh their network security defenses.
